# Архитектура системы Fitness Tracker

## 1. Общая архитектура

Система построена по принципу клиент-серверной архитектуры и состоит из следующих компонентов:

- PostgreSQL — реляционная база данных
- FastAPI — backend API
- SQLAlchemy — ORM для работы с БД
- Docker — контейнеризация приложения

Архитектура является трехслойной:
1. Слой данных
2. Слой бизнес-логики 
3. Слой представления 

---

## 2. Модель данных

### Основные сущности:

#### User
- id
- login
- first_name
- last_name
- email
- created_at

#### Exercise 
- id 
- name 
- description
- calories_per_minute

#### Workout 
- id 
- user_id 
- workout_date
- duration_minutes

#### WorkoutExercise 
- id 
- workout_id 
- exercise_id 
- sets
- reps
- duration_minutes

---

## 3. Связи между сущностями

- Один пользователь → много тренировок 
- Одна тренировка → много упражнений 
- Связь M:N реализована через таблицу workout_exercises

Схема:

User (1) —— (N) Workout  
Workout (1) —— (N) WorkoutExercise —— (1) Exercise  

---

## 4. Обоснование проектных решений

### Нормализация
База данных приведена к 3 нормальной форме:
- отсутствуют избыточные данные
- нет транзитивных зависимостей
- все атрибуты зависят только от первичного ключа

### Использование surrogate keys
Во всех таблицах используются искусственные ключи, что:
- упрощает связи
- ускоряет JOIN-операции

### Ограничения 
- PRIMARY KEY — уникальная идентификация записей
- FOREIGN KEY — целостность связей
- UNIQUE — предотвращение дубликатов
- CHECK — контроль корректности данных

---

## 5. Индексация

Используются следующие типы индексов:

### B-tree индексы
- users
- workouts
- workouts

Назначение:
- ускорение поиска
- оптимизация JOIN

### GIN индекс 
- для поиска по имени

Назначение:
- ускорение поиска по подстроке
- поддержка fuzzy search

## 6. Основные сценарии работы

### Пользователь:

- поиск по логину
- поиск по имени

### Работа с упражнениями:

### Работа с тренировками:
